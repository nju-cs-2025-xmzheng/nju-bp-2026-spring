## 基础部分

《杀戮尖塔》的核心机制是通过抽取、打出卡牌来与敌人战斗，进而通过关卡。在一关中，简化的流程如下：

1. 玩家开始自己的回合，获得一定能量，并从抽牌堆抽取一定数量卡牌作为手牌；
2. 玩家消耗能量打出手牌中的卡牌，被打出的卡牌进入弃牌堆；
3. 玩家结束回合，手牌中所有剩余的牌进入弃牌堆；
4. 敌人进入其回合并行动；
5. 敌人行动结束后再次进入玩家回合，回到第一步；若抽牌堆中数量不足，则将弃牌堆洗混后全部移入抽牌堆。

在基础部分中，我们实现基础的游戏逻辑和若干个关卡。具体的功能细节将会在对应章节中进行说明。

### 2. 战斗系统

在《杀戮尖塔》中，战斗的核心是玩家与敌人的数值博弈。任何复杂的战斗逻辑，底层都是对角色状态（如血量、能量、格挡）的读取和修改。在这一章中，我们将为游戏引入玩家和敌人的状态，并实现基础的伤害结算逻辑。

在这一章中，我们关注两个核心机制：**攻、防**。玩家和敌人之间可以相互造成伤害，也可以各自进行防御。防御会给角色提供格挡值，当一个角色受到伤害时，优先扣除格挡值，当格挡值为零时则扣除生命值。

#### 2.1 游戏信息

游戏的信息分为两种：

* **静态信息**：固定的数值，例如初始最大生命值、每回合能量、游戏初始卡牌等；
* **动态信息**：在游戏中发生变化的数值，例如当前生命值、剩余能量、各种牌堆状态等。

在这一章中，我们先关注最基础的几个信息：玩家和敌人的生命值与护甲。我们需要用变量来记录这些信息。因为这些信息会在很多地方用到（显示、攻击、判断结束等），为了方便，我们可以先把它们定义为**全局变量**。

目前我们先定义下面的信息，随着项目发展，信息会逐渐增多：

| 信息名称 | 类型 | 初始值 | 说明 |
| --- | --- | --- | --- |
| 玩家最大生命值 | 静态 | 80 | 玩家生命值的上限 |
| 玩家当前生命值 | 动态 | 80 | 玩家当前的存活状态，降至 0 时游戏失败 |
| 玩家当前格挡值 | 动态 | 0 | 用于抵消受到的伤害 |
| 玩家当前能量值 | 动态 | 3 | 打出卡牌需要消耗的资源 |
| 玩家初始能量值 | 静态 | 3 | 回合初能量回复至该值 |
| 敌人最大生命值 | 静态 | 40 | 敌人生命值的上限 |
| 敌人当前生命值 | 动态 | 40 | 敌人当前的存活状态，降至 0 时战斗胜利 |
| 敌人当前格挡值 | 动态 | 0 | 用于抵消受到的伤害 |

选择合适的变（常）量类型，定义所需的变量。在后续的章节中，你可以随时根据需求调整变量的组织。

#### 2.2 状态显示（HUD）

在真正的游戏中，玩家和敌人的状态通常会固定显示在屏幕的上方，这被称为 HUD。
编写一个专门用于显示双方状态的子程序（例如 `draw_hud`）。在这个子程序中，利用 `sts_io` 库的输出函数，将上一节中定义的变量按一定的格式排版打印出来。你可以自由设计排版，只要能清晰地展示出双方的血量（当前值/最大值）、格挡值以及玩家的能量即可。

在后续的开发中，我们只需要在每次调用 `sts_clear_screen()` 清空屏幕之后，紧接着调用一次 `draw_hud()`，就可以实现状态栏始终固定在屏幕最上方的视觉效果。

#### 2.3 伤害结算与防御

如前文所述，角色受到攻击时并不是直接扣除生命值。具体的伤害结算规则如下：

1. **优先扣除格挡**：当角色受到伤害时，首先比较伤害值与角色当前的格挡值。
2. **格挡充足**：如果当前的格挡值大于或等于受到的伤害值，则直接从格挡值中扣除相应的伤害值，生命值不发生变化。
3. **格挡击穿**：如果当前的格挡值小于受到的伤害值，则格挡值归零，并将剩余的伤害值从生命值中扣除。

请根据上述逻辑，分别编写两个子程序：`deal_damage_to_player(int damage)` 和 `deal_damage_to_enemy(int damage)`。

> 💡 **Tip**
> 生命值最低只能降至 0，不能出现负数。在扣除生命值后，记得加上一个判断，如果生命值小于 0，则将其强制赋值为 0。

#### 2.4 战斗测试循环

目前我们还没有引入卡牌系统，为了验证我们的状态显示和伤害结算逻辑是否正确，我们需要修改第一章中 "游戏正在进行中" 的那部分代码，将其变成一个临时的战斗测试菜单。

当玩家在主菜单选择 “进入游戏” 后，程序应该进入一个内部的战斗循环。在这个循环中，按以下顺序执行：

2. 打印几个测试选项供玩家选择：
* `0. 逃跑（返回主菜单）`
* `1. 攻击敌人（消耗 1 点能量，对敌人造成 6 点伤害）`
* `2. 进行防御（消耗 1 点能量，玩家增加 5 点格挡）`


3. 读取玩家的输入；
4. 根据输入执行对应的逻辑。**注意**：在执行攻击或防御前，必须先检查玩家的能量是否大于等于 1。如果能量不足，提示玩家并跳过本次操作；如果能量充足，则扣除 1 点能量并调用对应的函数。
5. 如果玩家选择 “逃跑”，则跳出当前的战斗循环，程序回到外层的主菜单循环。

> 💡 **Tip**
> 在后续部分，讲义可能不会再明确对子程序的组织进行提示。当你觉得一个子程序显得比较臃肿时，可以考虑重新组织子程序的结构。

#### 2.5 回合交替

《杀戮尖塔》是一个回合制游戏。当玩家的能量耗尽，或者不想再进行操作时，可以选择结束回合。此时会按序发生以下事情：

1. 玩家回合结束。
2. 敌人行动：敌人对玩家造成伤害（为了测试，我们暂时让敌人固定对玩家造成 8 点伤害）。
3. 敌人回合结束。
4. 玩家新回合开始：玩家的当前能量值恢复到满状态，格挡值清零。

#### 2.6 胜负判断

在任何时候，如果某个角色的血量降至 0，那么它就会死亡。玩家在一局中的胜利条件是敌人死亡，失败条件是玩家角色死亡。

### 3. 卡牌系统

《杀戮尖塔》中，玩家的行动通过卡牌实现，在讲义开头简要介绍了卡牌的运转流程。
在这一章中，我们将把前两章实现的固定操作转换成用数据描述的卡牌。

#### 3.1 卡牌的描述

每种卡牌都有一些用于描述其属性的信息：卡牌名、费用、说明以及被打出时的具体行为。目前我们认为这些属性都不会在游戏中发生改变。

卡牌可能有很多种，我们需要将这些信息通过一些方式存储下来。为了方便后续增添新的卡牌种类，我们可以将这些信息存在数组中。目前我们考虑下面三种卡牌：

| 卡牌名 | 费用 | 说明 |
| --- | --- | --- |
| 打击 | 1 | 对敌人造成 6 点伤害 |
| 防御 | 1 | 增加 5 点格挡 |
| 放血 | 0 | 对自己造成 3 点伤害，恢复 1 点能量 |

考虑设计三个数组 `card_name`, `card_cost` 和 `card_description`，其中 `card_name[0]` 表示第一种卡牌的名字，`card_name[1]` 表示第二种卡牌的名字，依此类推。

在完成上述数据的定义后，实现一个函数用于打印某一类卡牌的信息。你可以随时调整主函数的内容来测试新增的功能是否正确。

> 💡 **Tip**
> 字符串数组本质上是二维字符数组，由于不同的字符串长的可能不同，我们可以将每个维度的大小设置为一个足够大的数。
> 一个可能的写法如下：
> ```c
> char card_names[2][100] = {
>     "打击", "防御"
> };
> 
> ```
> 
> 
> 这里 `card_names` 共有两行，每一行有 100 列，列数可以视存储的字符串做调整。

#### 3.2 卡牌行为

在 3.1 节中，我们用数组表示了卡牌的一些属性，但是我们跳过了其被打出时的具体行为。
卡牌的行为可能很复杂，难以用一系列基础类型描述，所以我们可以通过一个函数或一段子程序来描述其行为。

如果想要同样使用数组存储子程序，我们可以利用函数指针，在这门课程中我们对函数指针不做要求。我们可以退而求其次，利用分支语句实现子程序的跳转。

实现函数 `play_card(int card_type)`，表示打出下标为 `card_type` 对应类型的卡牌，例如 `play_card(0)` 表示打出一张 “打击”，对敌人造成 6 点伤害。

> 💡 **Tip**
> 讲义中功能的实现方式并不唯一。如果你有更好的代码组织方式，可以自由选择实现方式，并视情况调整讲义后续内容的实现，最终实现功能基本一致即可。

#### 3.3 固定手牌

在 3.1 和 3.2 节中，我们定义了卡牌的信息。现在，我们让游戏逻辑真正使用这些卡牌。

在这一节中，我们先不考虑复杂的抽牌和弃牌堆机制。我们假设玩家的手牌是固定的 5 张牌：两张 “打击”、两张 “防御” 和一张 “放血”。并且，为了简化当前的逻辑，卡牌被打出后不会消失，也不会进入弃牌堆，玩家只要有足够的能量，就可以反复打出这些牌。

我们需要一种方法来记录玩家的手牌。目前我们的手牌固定为 5 张，可以使用一个长度为 5 的数组来记录手牌。
手牌的组织同样有不同的策略，这里给出两种方式，思考各自的利弊并选择一种进行实现：

* 利用一个数组，存储卡牌类型对应的下标，例如 `{0, 0, 1, 1, 2}` 即表示 “打击”“打击”“防御”“防御 Grandma”“放血”；
* 利用若干个数组，仿照卡牌类型的方式分别存储每张手牌的描述，如果需要则另外处理被打出时的行为。

#### 3.4 自动化测试

在第二章中我们提到，对于早期的小游戏，最好的测试方式就是上手玩一玩，这能帮我们直观地检查界面和流程是否顺畅。
但是，随着游戏逻辑越来越复杂，如果每次修改代码后，都要手动运行游戏、一层层输入选项去验证某一张卡牌的数值对不对，效率会变得极其低下。

因此，对于纯粹的 “数值逻辑”，我们希望程序能够 “自己测试自己”，这就是自动化测试的基础思想。
自动化测试的核心逻辑非常简单，通常分为三步：

1. **设置初始状态**：人为地把变量设定为一个已知的值；
2. **执行目标动作**：调用你想要测试的函数；
3. **检查执行结果**：判断变量变化后的值是否符合预期。

课程中讲解过 `<assert.h>` 库中的 `assert` 宏。它接受一个条件表达式，如果条件为假，程序会直接报错并终止，我们可以利用它来进行第三步的检查。

尝试调整你的程序组织，并编写一个子程序 `test_logic()`。在这个子程序中，我们不依赖任何用户输入，直接用代码模拟游戏过程。例如，测试 ”放血“ 卡牌的逻辑如下：

```c
#include <assert.h>

void test_logic(void) {
    // 设置初始状态
    player_hp = 80;
    player_energy = 0;
    
    // 执行目标动作: 打出“放血”(假设其卡牌种类下标为2)
    play_card(2);
    
    // 检查执行结果
    assert(player_hp == 77);       // 预期受到3点伤害
    assert(player_energy == 1);   // 预期恢复1点能量
    
    /* 测试通过 */
}

```

> 💡 **Tip**
> 要让程序能够 “自己测试自己”，所有被测试的核心逻辑（如 `play_card`）就不能被输入卡住。如果需要测试的函数里包含了读入玩家输入的函数，那么就无法达到自动化测试的目的。同样，我们通常也不在这些核心逻辑中包含输出，防止干扰自动化测试的表现（测试输入输出本身时除外）。尝试以 “可自动测试” 为目标组织你的代码。

#### 5.6 随机化封装

接下来我们来实现随机洗牌逻辑。在课程中我们介绍了 `rand` 和 `srand` 函数，我们利用它们实现一个随机洗牌算法。

我们知道，在使用 `rand` 生成随机数前，需要调用 `srand` 设置随机种子。但在大型项目中，很容易忘记在主函数开头调用它，或者不小心在循环里多次调用导致随机数不随机。

我们可以设计一个 `init_rand()` 函数，利用静态局部变量来记录是否已经初始化，若没有初始化则调用 `srand` 设置随机种子。接着，设计一个辅助函数 `randint(int min, int max)`，在它的开头调用 `init_rand()`，然后返回一个在 `min` 到 `max` 范围内的随机整数。
你可以在实现这两个函数后写一些简单的测试来验证正确性。

#### 5.7 随机洗牌

打乱一个数组最经典且高效的算法是 **Fisher-Yates 洗牌算法**。它的核心思想是从数组的最后一个元素开始，随机与前面的某个元素交换位置。

假设牌堆的长度为 `length`，算法步骤如下：

1. 让一个循环变量 `i` 从 `length - 1` 开始，递减到 1；
2. 在每次循环中，调用 `randint(0, i)` 生成一个随机下标 `j`；
3. 将数组中下标为 `i` 的元素与下标为 `j` 的元素进行交换。

实现一个 `shuffle_pile()` 子程序用于打乱抽牌堆。在每局游戏开始时（牌库复制到抽牌堆后），以及抽牌堆耗尽、弃牌堆转移到抽牌堆后，调用这个洗牌函数。

#### 5.8 随机化测试地狱（可选）

引入随机性后，自动化测试会遇到一个巨大的挑战：你无法预测下一次抽到的是什么牌，也就无法用 `assert` 写死预期的状态。
一种常见的解决方案是：**让随机算法变得“可控”**。

我们可以利用条件编译设计一个测试模式的随机数生成器。在测试模式下，不要使用 `time(NULL)` 作为种子，而是传入一个固定的常数（例如 `srand(42)`）。因为固定种子的伪随机数序列是固定的，这样洗牌结果就会变得可以预测，从而允许你重新编写涉及抽牌逻辑的自动化测试。
利用条件编译，让你的自动化测试在包含随机化逻辑时依然能运行。

### 6. 敌人意图与行动

在前面的章节中，我们完善了玩家的卡牌系统。但是，目前敌人每回合只会造成固定的伤害，战斗体验比较单调。

在《杀戮尖塔》中，敌人的行为是多变的，且一个重要的机制是**敌人下回合的意图会显示给玩家**，玩家可以预知敌人下回合要做什么，从而决定本回合的行动。
在这一个章节中，我们将会实现敌人的不同的行为和意图显示。

#### 6.1 记忆

我们先实现一个简单的行为模式：

* **第一回合**：为自己增加 6 点格挡；
* **第二回合**：对玩家造成 8 点伤害；
* **第三回合**：对玩家造成 5 点伤害并为自己增加 10 点格挡；
* **第四回合**：重复第一回合的操作，依此循环。

要让敌人能够按照规律行动，敌人就需要记忆自己正在进行第几回合。设计变量存储敌人当前的回合数，并在正确的时机更新回合数。在战斗界面中，输出敌人将要（或正在）进行的回合数。

#### 6.2 实现循环行为与意图显示

抽象出一个名为 `enemy_turn` 的子程序（如果你先前没有的话），用于处理敌人回合的行为。
另外，用类似的逻辑，实现 `show_enemy_intent` 子程序，用于打印敌人意图。
将两段子程序整合到战斗循环中，将 6.1 中的敌人回合数修改为敌人意图。

> 💡 **Tip**
> 你也可以尝试设计 `get_enemy_intent` 子程序，使其返回一个表示意图的字符串，但是你需要尤其注意内存管理问题，确保字符数组被正确处理。

#### 6.3 更复杂的行为

一直循环一系列行为仍显得有些单调，在这一小节中，我们为敌人设计一个 “二阶段”：

* 一阶段与 6.1 中描述的行为相同；
* 在任意血量降至一半以下（含一半）时进入二阶段；
* 进入二阶段时，无论先前意图为何，下一轮都变为对玩家造成 16 点伤害；
* 在二阶段中（不包含进入二阶段的回合），行为在 “格挡 10 点” 和 “造成 12 点伤害” 中交替。

**注意**：如果玩家的操作让敌人进入了二阶段，战斗界面中的意图应该立刻更新。

此时单纯记录回合数就不够用了，我们需要记录更多信息来描述敌人的状态，例如敌人目前所处的阶段。在处理一阶段到二阶段的过渡时，我们可以引入一个中间阶段来统一行为。

这种通过存储不同的状态在不同行为间切换的设计称为**状态机**。依赖一些状态的存储与分支判断，我们可以设计出各种各样的敌人，这也是早期游戏以及现在许多游戏中 NPC 实现的基础。
设计合适的状态，组织逻辑完成敌人行为的实现，并同样实现意图显示。

## 扩展部分

扩展部分有两个大方向：功能扩展和代码重构。你可以任选一个方向。

### 方向一：功能扩展

让你的游戏内容变得更加丰富。如果你选择这个方向，你只需要实现下面小方向的一种。

1. **状态效果 (Buff / Debuff)**
实现经典的战斗状态效果，并设计特定的卡牌来触发它们：
* **易伤**：受到的攻击伤害增加 50%。
* **虚弱**：造成的攻击伤害减少 25%。
* **中毒**：每回合失去等同于层数的生命值，随后层数减 1。


2. **多敌人与“小怪”机制**
打破单挑的限制，实现一对多的战斗：
* 设计单关卡多敌人，允许玩家在打出攻击牌时，输入目标编号选择攻击哪个敌人。
* 实现 “小怪” 机制：有些敌人是首领，有些是小怪。当场上所有的非小怪（首领）均死亡时，剩余的小怪会自动逃跑/死亡，直接判定战斗胜利。


3. **连续关卡与卡牌奖励**
设计真实的闯关体验：
* 制作三个连续的关卡，每一关面对不同的敌人或敌人组合。
* 通过一个关卡后，玩家保留当前状态进入奖励环节。
* **卡牌奖励**：随机给出 3 张卡牌，玩家选择一张永久加入牌库，随后进入下一关。


4. **消耗与虚无机制**
引入第四个卡牌存放区——**消耗牌堆**：
* **消耗**：带有此词条的卡牌被打出后，移入消耗牌堆而不是弃牌堆，本局战斗无法再次使用。
* **虚无**：带有此词条的卡牌如果在回合结束时仍在手牌中，会被强制移入消耗牌堆。


5. **遗物系统**
设计能在整局游戏中提供被动增益的遗物系统。不同的遗物需要有不同的触发机制：
* 例如遗物 A：在战斗的第三回合对敌人造成伤害；
* 例如遗物 B：在每局游戏中，如果玩家连续打出 3 张特定类型的卡牌，可以额外抽取一张牌。



### 方向二：代码重构（较难）

如果你对软件架构和代码的优雅程度有更高追求，可以尝试向 “数据驱动” 的方向重构你的代码。如果选择这个方向，你需要尝试完成所有重构目标。

随着功能的增加，如果仅用全局变量和散落的 `switch-case`，代码会迅速变得臃肿。你可以尝试：

* **完全的结构体化**：将玩家、敌人甚至整个战局的状态全部封装进不同的 `struct` 中，消除所有全局变量。
* **函数指针与数据驱动**：在卡牌结构体中存入函数指针，把每张卡牌的具体效果写成独立的子程序，彻底消除 `play_card` 中的分支语句。
* **多模块化**：运用课程中学习的头文件知识，将不同功能的代码拆分至多个 `.c` 和 `.h` 文件中，规范模块间的接口。
